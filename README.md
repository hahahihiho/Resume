- ETF와 같이 여러 자산들을 한번에 분산투자할 수 있는 smart contract를 만들려는 시도도 했었지만 여러가지 제약과 생길 문제점들을 커버하기에 많은 연산량과 가스비가 구현된 복잡도에 비해 효율적이지도, 단순하지도 않았습니다.

- WemixFi 를 런칭했습니다. 코인의 가격과 거래량 데이터를 블록체인네트워크에서 가져와 차트데이터로 만드는 일을 했습니다. 차트를 보여주는 방식에 따라 어떤타이밍의, 어떻게 계산/측정된 가격을 쓸 것인지가 중요했고, 특히나 변동성이 큰 암호화폐 시장에서 중요한 가격 변동성을 다 보여주기가 쉽지 않아 타협점에 대한 논의가 많이 필요했습니다. 가격과 거래량을 추적하기위해 swap event와 tx을 following 하는것도 필요해 Queue(RabbitMQ)를 사용하기도 했고, 어떤 시간 구간을 요청하던 차트데이터를 실시간으로 보여주고 싶어 효율적인 시계열 데이터를 다루는 DB(influxDB)도 시도했었으나 요청시 마다 raw데이터에 계산하는 것은 부하가 커서 보여줄 시간구간을 정책적으로 정하고 미리 계산해두어야 한다는 결론을 냈습니다. 최대한 효율적으로 실시간 데이터를 보여주기 위해 마지막 데이터만 효율적으로 업데이트 하는 로직이나 데이터 업데이트 타이밍 그리고 cache를 잘 연동하기 위한 로직을 많이 고민했던 시기였습니다.

- AI를 Agent로 쓰며 서로 소통하여 결과를 내는 코드를 만드는 시도를 했습니다. 그 과정에서 CrewAI라는 프레임워크도 사용했었습니다. 하지만 AI가 모든 내용을 반영하지 못하는 부분과 수정을 요청했을 때 원하는 방향으로 가지 않는 경우가 많아, 작은 prototype 이상으로 큰 프로젝트를 자동화 하기엔 어렵단 결론을 냈습니다.

- AI툴을 만들면서 중간중간 사람 개입의 필요성과 완전한 자율화보다는 정해진 동작을 수행하는 Tool의 필요성을 느끼고 API를 연동하는 MCP을 만들었습니다. MCP가 헷갈리지 않고 할 수 있는 동작의 수에 제한이 있었기에 API 마다 동작을 만들지, API설명서를 만들고 API요청을 만들어 설명서를 읽어 요청을 보내게 할지 등 다양한 구조에 대해 고민을 했었습니다.

- 회사 문서를 잘 가져다 쓸 수 있는 RAG에 대해 고민했습니다. 결과적으로는 langchain으로 단어 수 기준으로 잘라 vector DB에 넣고 요청시 불러오는 기본적인 방법을 구현하였지만 일반화된 문서의 목차를 만들고 더 정밀한 문서구조와 검색구조에 대해 고민하던 시기였습니다.
