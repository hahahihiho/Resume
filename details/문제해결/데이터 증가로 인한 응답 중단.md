# 데이터 증가로 인한 응답 중단 문제 해결

#### 📌 문제 상황 및 원인 분석

블록체인 네트워크에서 이벤트 데이터를 수집하는 과정에서, 이벤트 발생량이 급증할 경우, 아래와 같은 현상이 반복적으로 발생.

- 데이터 수집 중단
- 데이터 누락
- 처리 순서가 보장되지 않아 정합성이 깨짐

문제 원인을 분석하기 위해 로그를 기반으로 병목 구간을 추적한 결과, 블록체인 네트워크에 요청한 이벤트가 많을 시 응답에 문제가 있었다.

부하의 가중으로 생긴 문제라 아래의 문제점들을 해결해야 할 것이라 생각했다
- 특정 시기에 간헐적으로 많은 transaction이 발생한 경우 네트워크에서 데이터를 가져오다 멈추는 현상
- 재처리 로직의 부족으로 데이터 수집중단과 유실이 발생하는 구조
- 구조적 이점을 위해 중복된 데이터 여러번 요청(네트워크 부하 가중)


---

#### 🛠 해결 방법 및 선택 이유

처음엔 요청하는 batch 사이즈를 줄여 처리하는 형태로 단순 수정을 했지만 단위가 작어지니 또 다른 문제들이 발생하였다.

- 요청 횟수가 늘어나며 중복된 요청이 많아져 네트워크에 부하를 가중
- 마지막 업데이트가 오래되었을 시, 마지막부터 최신까지의 범위가 너무 커지는 점

그래서 아래와 같은 해결책을 생각하였다.

- 중복된 요청은 caching 해두어 네트워크 부담을 줄이고 속도 향상
- 처리한 지점(블럭)까지 처리 완료 여부 기록

그러면서 historical data를 수집하는 것과 잘 합치면 통신 부하를 줄일 수 있지 않을까 하는 생각에 기존 polling 방식의 요청에서 네트워크의 블록생성을 listening 하여 데이터를 받아와 파싱하는 구조로 변경하기로 했다. (하지만 이는 오히려 데이터 파싱을 하는 서버에 부담으로 작용하게 되는데..)

그래서 최종적으로 아래와 같은 구조가 설계되었다.
- 네트워크의 블록생성을 listening
- 블록정보를 가져와서 queue에 저장
- queue에 저장된 정보를 컨슈머들이 가져가서 파싱하고 historical data와 차트 및 통계데이터를 생성
- 처리한 지점(블럭)까지 처리 완료 여부 기록
- 그리고 파싱 과정에서 서버의 부담이 늘어났지만, container와 k8s를 사용하여 부하를 유연하게 관리.

그리고 queue와 분산처리를 사용하게 되며 생기는 ordering 문제는 따로 Buffer에 보내어 순차적으로 관리해 주었다.  그리고 Buffer 가 일정 이상으로 쌓이면 monitoring 알람을 보냈다(telegram)

그 외에도 문제 지점을 알기 쉽게하는 logging 카테고리화와 queue의 흐름과 데이터 생성로직에 대한 test 코드도 추가했다.
