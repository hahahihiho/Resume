# 시계열의 실시간성 문제

#### 📌 문제 상황 및 원인 분석

시계열 데이터를 보여줄 때, 원하는 시간 간격(1주, 1일, 4시간, 5분 등)과 기간을 입력하면 데이터를 동적으로 가공하여 차트로 제공하는 기능을 구현했다.

초기에는 단순 선형 그래프였는데, 가격이 급변하는 구간이 제대로 표시되지 않는 문제가 발생했다.

이를 해결하기 위해

- OHLC(Open, High, Low, Close) 기반의 캔들 차트
- 거래량 누적 데이터

를 함께 제공하도록 구조를 확장했는데 이 과정에서 SQL Query가 복잡해져 DB 부하가 증가했다

---

#### 🛠 해결 방법 및 선택 이유

DB 의 부하를 줄이기 위해 아래 방법들을 검토하고 적용해 봤는데
- 적당히 처리한 Query와 서버에서의 추가 가공 -> 복잡도와 서버 부하 증가
- InfluxDB와 같은 시계열 데이터를 처리하는 DB도 적용 -> 여전히 한계점 존재
- API 요청 시마다 데이터를 생성하고 캐싱하는 방식 
	-> 요청이 없는 구간의 데이터는 갱신되지 않아 최신성이 보장되지 않고
	-> 특정 시점에 요청이 몰릴 경우 여전히 부하가 집중되는 문제가 있었다.

요청 시점에 실시간으로 계산하는 구조는 한계가 있다고 판단하고, 데이터를 미리 가공해두고 요청 시엔 조회만 하는 방식이 더 안정적이라 결론내렸다.

그래서

- Frontend팀과 협업하여 요청 가능한 시간 간격을 제한하고
- 주기적으로 OHLC 및 누적 데이터를 미리 계산하여 저장하는 배치 프로세스를 구현했다.

---

#### 🙋‍♂️ 담당 역할

- 시계열 데이터 구조(OHLC, 누적값)에 따른 병목 원인 분석
- 다양한 대안(InfluxDB, 캐싱 구조 등) 검토 및 한계 도출
- API 구조 개선 및 Frontend와의 협업 주도
- 사전 집계 로직 및 스케줄링 파이프라인 설계 및 구현

---

#### 🚀 임팩트 및 성과

- 요청 시 연산 제거를 통해 응답 속도 개선 및 타임아웃 문제 해소
- DB 및 서버 리소스 사용량 감소
- 확장 가능한 처리 구조
